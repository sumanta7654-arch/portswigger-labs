Lab Description

This lab has a brute-force protection mechanism that blocks an IP address after three consecutive failed login attempts. However, this protection is flawed and can be bypassed by strategically interleaving valid login attempts.

The goal is to brute-force the user carlos's password by resetting the failure counter before the block kicks in.

Mindset & Strategy

The core logic of this vulnerability is a Counter Reset Flaw. The server tracks consecutive failures. If we fail twice and then succeed once with our own account (wiener:peter), the "consecutive failure" count resets to zero.

Since X-Forwarded-For is not working here, we must act like a stealthy attacker who resets the alarm just before it goes off. By using a 2:1 ratio (2 attempts on Carlos, 1 successful login for us), we can bypass the IP block indefinitely.

Step-By-Step-Execution

1. Intercept & Analyze: Log in with wiener:peter and capture the POST request. Send it to Repeater and observe that after 3 wrong attempts, the server blocks the IP. Notice that headers like X-Forwarded-For are ignored by the WAF/Server.

2. Prepare the Payload List: We need to create a custom list. For every two passwords from the Carlos list, we must insert our own valid credentials (wiener:peter) as the third attempt.

3. Intruder Configuration: Set the attack type to Pitchfork. Set Payload 1 on the username parameter and Payload 2 on the password parameter.

4. Setting Payloads:
• Payload 1: carlos, carlos, wiener (repeatedly).
• Payload 2: pass1, pass2, peter (repeatedly from the list).

5. Resource Pool: This is critical. Go to the Resource Pool tab and set Maximum concurrent requests to 1. If we send requests in parallel, the sequence will break, and we will get blocked.

6. Start Attack: Start the attack and watch the responses. Every third request (our login) should be a 302 or 200, keeping the connection open.

7. Identify the Password: Sort the results by the carlos username. Look for a 302 Redirect status code that isn't our own account. That is the correct password for Carlos.

8. Final Step: Right-click the successful request, select "Show response in browser," and log in to solve the lab.

Technical Explanation

The vulnerability lies in the State Management of the brute-force protection. Instead of tracking total failed attempts over a window of time, the server only tracks consecutive failures. By injecting a successful login into the stream of failed attempts, we "cleanse" the session state, effectively blinding the security monitor to our brute-force attack.

How To Fix

°      Global Rate Limiting: Implement rate limiting based on the total number of attempts per IP/Account within a fixed time window (e.g., 5 mins), regardless of successful logins in between.

°      Progressive Delays: Introduce an exponential backoff (increasing wait time) for every failed attempt to slow down automated scripts.

°      Account Lockout: Use a robust lockout mechanism that requires MFA or CAPTCHA once a threshold is reached.
