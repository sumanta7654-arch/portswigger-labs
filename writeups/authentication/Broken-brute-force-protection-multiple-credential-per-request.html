Authentication | Practitioner

Lab Description

This lab has a brute-force protection mechanism that limits the number of failed login attempts. However, it is vulnerable to a specific type of attack where multiple passwords can be submitted in a single request.

Goal: Bypass the rate limit by wrapping the entire password list into a JSON array and submitting it for the user carlos.

Mindset & Strategy

Standard brute-force protection counts how many requests are sent from an IP or to an account. If we send 100 requests with 1 password each, we get blocked. But what if we send 1 request with 100 passwords?

This is a Logic Flaw in the authentication handler. If the backend iterates through an array of passwords within a single session/request, the rate limiter (which counts requests) will only see "one" attempt, while we have actually tested the entire wordlist.

Step-By-Step-Execution

1. Capture & Analyze: Log in with wiener:peter and capture the POST request in Burp Suite. Send it to Repeater. Notice that after a few failed attempts, the server returns an error message regarding attempt limits.

2. Test for Array Support: Try changing the password value from a string to an array: "password":["123456","password"]. If the server doesn't throw a 400 Bad Request, it likely supports multi-credential processing.

3. Prepare the Payload: Copy the password list from the lab description. We need to format these into a JSON array.
Example: ["pass1", "pass2", "pass3", ...]. You can use a simple Python script or a text editor to wrap each password in quotes and add commas.

4. Execute the Attack: In Repeater, set the username to carlos and replace the password string with your full JSON array. Send the request.

5. Verify Success: Check the response. You should see a 302 Found or a successful login message. Right-click and select "Show response in browser" to confirm you are logged in as Carlos.

6. Lab Solved: By exploiting this single-request multi-password vulnerability, we effectively bypassed the IP-based rate limiting entirely.

Technical Explanation

The vulnerability occurs because the application's Rate Limiter and Authentication Engine are not synchronized. The Rate Limiter counts "HTTP Requests," whereas the Authentication Engine processes "Credentials." By submitting an array, we cause the engine to perform a foreach loop on the passwords, while the Rate Limiter only increments its counter by one for the entire batch.

How To Fix

°      Input Validation: Strictly enforce that the password field must be a single string, not an array or object.

°      Atomic Rate Limiting: Ensure that the rate limit counter increments for every single password validation attempt, even if they arrive in a single request.

°      Fail-Fast Logic: If the application receives unexpected data types (like an array for a password), it should immediately terminate the session and flag the activity as suspicious.
