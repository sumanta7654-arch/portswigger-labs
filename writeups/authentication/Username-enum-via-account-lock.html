Lab Description

This lab is vulnerable to username enumeration via account lockout. By monitoring how the application reacts after multiple failed attempts, we can distinguish between a valid and an invalid username.

Goal: Enumerate a valid username, identify when it gets locked, and then brute-force the password for that specific user.

Mindset & Strategy

Most secure applications lock an account after several failed attempts. If an application locks valid_user but doesn't care how many times we try with invalid_user, it reveals which account actually exists in the system.

Our strategy is to trigger this lockout purposefully. We will use Cluster Bomb to try multiple passwords against many usernames and look for a specific error message: "You have made too many incorrect login attempts. Please try again in 1 minute(s)."

Step-By-Step-Execution

1. Intercept & Investigate: Open Burp Suite, try a random login, and capture the POST request. Send it to Repeater. Notice that a few failed attempts don't immediately show a lockout for random users.

2. Intruder Setup (Phase 1): Send the request to Intruder and select the Cluster Bomb attack type.

3. Payload Positions: Set Payload 1 on the username parameter and Payload 2 on the password parameter.

4. Payload Lists:
• Payload 1: Paste the username list from the lab description.
• Payload 2: Enter 5 random, unique passwords (e.g., 123, 456, etc.). This ensures we hit the lockout threshold for each user.

5. Identify Valid User: Start the attack and sort by Length. You will see some responses are longer than others. Upon inspection, these longer responses contain the message: "You have made too many incorrect login attempts. Please try again in 1 minute(s)."

6. Observation: Only the valid username (e.g., azure or similar) will trigger this message after the 5th attempt. Now we have our target username.

7. Intruder Setup (Phase 2): Clear the previous settings. Switch the attack type to Sniper. Fix the username to the valid one we just found and set the payload on the password parameter.

8. Password Brute-Force: Paste the password list from the description into the payload configuration and start the attack.

9. Analyze Sniper Results: Sort the results by Length again. One result will stand out because it won't have the "invalid password" or "too many attempts" warning. This is our successful login.

10. Manual Login: Take the identified username and password, go to the real login page, and enter them manually.

11. Success: Once logged in, the lab status will change to solved. You have successfully enumerated a user through defensive side-effects!

Technical Explanation

The vulnerability exists because the Account Lockout mechanism is applied only to existing users. When the backend checks a login, it first verifies the username. If it exists, it increments a 'failed attempt' counter. If it reaches the limit, it changes the response message. Since this doesn't happen for non-existent users, the difference in response (length and content) leaks the validity of the username.

How To Fix

°      Generic Lockout: The application should ideally mimic a lockout response even for invalid usernames after a certain number of attempts from a specific IP.

°      Consistent Messaging: Use a generic message like "Invalid credentials or account temporarily disabled" for all scenarios.

°      IP-Based Rate Limiting: Implement rate limiting at the IP level to prevent an attacker from systematically testing an entire username list.
