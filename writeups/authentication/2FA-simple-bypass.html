Authentication | Easy

Lab Description

This lab's two-factor authentication can be bypassed. You have already obtained a valid username and password, but do not have access to the user's 2FA verification code. To solve the lab, access Carlos's account page.

°      Your credentials: wiener:peter

°      Victim's credentials carlos:montoya

Mindset & Strategy

In a secure environment, authentication should be a strict, multi-step process where Step B (2FA) cannot be completed without Step A (Username/Password), and the final session is only granted after both are verified.

Our strategy here is to test the State Management of the login flow. We want to see if the application "remembers" that we are partially authenticated after entering the password and if it incorrectly grants access to the user's dashboard if we simply skip the 2FA URL. If the backend assumes that reaching the /my-account page is proof of identity without verifying the 2FA completion, the entire security layer is bypassed.

Step-By-Step-Execution

1. Analyze the Flow: Open the lab and log in with the victim's credentials (carlos:montoya).

2. Observe the URL: After submitting the password, you are redirected to the 2FA entry page (e.g., /login2). At this point, do not enter a code.

3. Test the Bypass: Instead of completing the 2FA, manually change the URL in your browser’s address bar to /my-account (or simply click the "Home" or "Account" link if available).

4. Verify Access: Because the backend has already created a session for "carlos" after the password check and fails to verify if the 2FA was successful, you will find yourself logged into his account.

5. Result: The 2FA step is completely bypassed, and the lab is solved.

Technical Explanation

The vulnerability lies in Incomplete Session Validation. The application logic is flawed because it treats the "Logged In" state as a single boolean (True/False) based only on the first factor.

In a vulnerable system:

°      Input Password -> Session set to "Authorized".

°      Redirect to 2FA -> User is expected to go here, but the session is already active..

In a secure system:

°      Input Password -> Session set to "Pre-Auth" (Limited access).

°      Input 2FA -> Session upgraded to "Fully-Auth" (Full access).

Conclusion

A 2FA system is only as strong as its enforcement. If the second factor is merely a "suggestion" in the UI rather than a mandatory check in the backend, it provides zero additional security. This lab reminds us that as security researchers, we should always try to "jump" or "skip" steps in a logical process to see how the server reacts.

How To Fix

°      Mandatory Step Verification: The server must check the user's authentication status at every sensitive endpoint. Access to /my-account should be denied unless a "2FA_Verified" flag is present in the session.

°      State-Based Authentication: Use a multi-stage authentication state. Users should remain in a "restricted" state where they can only access the 2FA submission page until the correct code is provided.

°      Session Revocation: If a user tries to access protected resources before completing 2FA, the system should ideally terminate the partial session and force a re-login.
