Authentication | Easy

Lab Description

This lab's password reset functionality is vulnerable. To solve the lab, reset Carlos's password then log in and access his "My account" page.

°      Your credentials: wiener:peter

°      Victim's username: carlos

Mindset & Strategy

The password reset process is one of the most sensitive parts of any web application. Typically, a secure flow requires a unique, one-time token linked to a specific user. However, in this lab, we are looking for a Logic Flaw.

Our goal is to analyze the final request that actually changes the password. We want to see if the server "trusts" the username provided in the request body instead of verifying the reset token. If the application doesn't double-check that the person holding the token is actually the person whose password is being changed, we can exploit this "trust" to reset anyone's account—including the victim, Carlos.

Step-By-Step-Execution

1. Analyze the Normal Flow: First, understand how a legitimate reset works. Go to the login page, click "Forgot Password," and enter your own username (wiener).

2. Access the Token: Open the Email Client from the lab header, find the password reset email, and click the link.

3. Intercept the Final Step: On the password reset page, enter a new password (e.g., test123). Before clicking submit, make sure Burp Suite Intercept is ON.

4. Manipulate the Logic: * Capture the POST /password-reset request.

°      Send this request to Burp Repeater.

°      Notice the parameters: there is likely a temp-forgot-password-token and a username=wiener.

°      The Exploit: Simply change the username value from wiener to carlos.

5. Execute the Attack: Send the modified request in Repeater. The server should return a 302 Found or a success message.

6. Verify the Hack: Go back to the login page and try logging in as carlos with your new password (test123).

7. Result: You are logged in as Carlos, and the lab is solved.

Technical Explanation

The vulnerability occurs because the backend performs insufficient validation. The server correctly issues a token to wiener, but when the final "change password" request is sent, the backend only checks if the token is valid. It fails to verify if the username parameter matches the user to whom the token was originally issued. This allows an attacker to use their own valid token to change the password of any other user in the database.

Conclusion

This lab is a powerful reminder that technical security (like having a complex reset token) is useless if the logic behind it is broken. As hackers, we must always look for "User-Controlled Inputs" in sensitive requests. If a username or ID is present in a request that performs a sensitive action, we should always try to swap it with another user's details to see if the server is blindly trusting our input.

How To Fix

To secure the password reset flow, developers should implement these practices:

°      Token-to-User Mapping: The database must strictly associate a reset token with a specific user ID. When the reset request is received, the server should ignore any "username" parameter sent by the user and instead change the password for the account linked to that specific token.

°      Hidden Identifiers: Never rely on user-provided parameters like username or email in the final step of a sensitive process.

°      One-Time Use: Ensure that once a token is used, it is immediately invalidated and cannot be reused for any other account.
