Authentication | Expert

Lab Description

This lab's two-factor authentication is vulnerable to brute-forcing. However, the application has a security mechanism: after two failed MFA attempts, the user is logged out, and the session is invalidated. This prevents a standard automated attack.

Goal: Use Burp Suite Macros to automate the login process and brute-force the 4-digit MFA code for the user carlos.

Mindset & Strategy

Brute-forcing a 4-digit code (0000-9999) is usually easy, but here the server resets our state every two tries. To bypass this, we need a Macro. A macro is a sequence of requests that Burp Suite runs automatically before every attack request.

The strategy is to tell Burp: "Before you try an MFA code, first go to the login page, submit Carlos's credentials, and get a fresh CSRF token and MFA session." By setting the Resource Pool to a single thread, we ensure the sequence never breaks.

Step-By-Step-Execution

1. Intercept the Flow: Turn on Burp Intercept. Log in with carlos:montoya. You will be sent to the MFA page (/login2). Enter any 4-digit code and capture that request. Also, ensure you have captured the initial GET /login and POST /login requests in your HTTP history.

2. Create the Macro: Go to Settings > Sessions > Macros. Click Add. Select the following three requests in order:
• GET /login (To get a fresh CSRF token)
• POST /login (To submit credentials)
• GET /login2 (To land on the MFA entry page)

3. Configure Session Handling: Under Settings > Sessions > Session Handling Rules, click Add.
• In Scope, select "Intruder" and "Include all URLs".
• In Details, add a rule action: Run a macro. Select the macro you just created.

4. Update Parameters: In the Macro configuration, ensure Burp is set to update the parameters (like CSRF tokens) from the previous responses automatically. This is crucial for the POST /login to succeed.

5. Set Up Intruder: Send the POST /login2 request (the one with the MFA code) to Intruder.
• Positions: Highlight the 4-digit MFA code as the payload position.
• Payloads: Select "Numbers". Set range from 0 to 9999. Set "Min integer digits" and "Max integer digits" to 4 to ensure it tries 0001, 0002, etc.

6. Resource Pool (Critical): Go to the Resource Pool tab. Create a new pool and set Maximum concurrent requests to 1. If you use multiple threads, the macro sessions will overlap and fail.

7. Launch the Attack: Start the attack. It will be slow because Burp is performing 4 requests for every single code attempt. Monitor the Status Code or Length columns.

8. Solve: When you find a 302 Found status code, that is the correct MFA key. Right-click that request, select "Show response in browser," and paste the URL into your browser to log in as Carlos and solve the lab.

Technical Explanation

This is an Expert-level Authentication Bypass because it defeats "Rate Limiting" and "Session Termination" defenses. The server correctly identifies an attack and kills the session, but it fails to implement a permanent lockout or a CAPTCHA. Because the application logic allows a user to immediately re-authenticate after a session reset, we can programmatically simulate a legitimate user login for every single brute-force attempt using Burp's session handling engine.

How To Fix

°      Progressive Delays: Implement increasing time delays after failed MFA attempts (e.g., 1 min, 5 min, 30 min).

°      Account Lockout: Lock the account after a certain number of failed MFA attempts, regardless of whether the user re-logs in with their password.

°      Notification: Notify the user via email or SMS when multiple MFA attempts are detected, allowing them to secure their account.
