Authentication | Practitioner

Lab Description

This lab is vulnerable to username enumeration based on a noticeable difference in response times. It also has a brute-force protection mechanism that blocks IPs after too many failed attempts.

To solve the lab, we must bypass the rate limit, enumerate a valid username using response timing, and then brute-force that user's password.

Mindset & Strategy

When an application takes longer to process a specific request, it reveals a "leak" in the backend logic. In this case, the server likely hashes the password only if the username exists in the database. Since password hashing is a CPU-intensive task, a valid username will trigger a slower response than an invalid one.

However, we face a barrier: **Rate Limiting**. To overcome this, we will use the X-Forwarded-For header to spoof our IP address for every request, making the server think each attempt is coming from a different user.

Step-By-Step-Execution

1. Analyze Login Flow: Login with wiener:peter and capture the POST request. Send it to Repeater. Notice that after 3 failed attempts, the application starts blocking requests.

2. Bypass Rate Limit: Add the X-Forwarded-For: 1 header. After sending, you'll see the attempt limit resets. We will automate this using a dynamic IP for every request.

3. Username Enumeration (Pitchfork): Send the request to Intruder and set the attack type to Pitchfork. Set two payload positions: one for the last digit of the IP in X-Forwarded-For: 1.1.1.§1§ and one for the username=§wiener§ parameter.

4. Payload Configuration: Payload 1 is a "Numbers" type (1 to 100). Payload 2 is the list of usernames provided in the lab description. Start the attack.

5. Identify Valid Username: Sort the results by the "Response completed" column (time). Look for a request that takes significantly longer (double or triple the usual time). In this case, the username announce triggered a delayed response.

6. Password Brute-Force: Keep the X-Forwarded-For IP rotation to avoid being blocked. Change the attack to target the password parameter using the list of passwords provided.

7. Final Exploit: Run the attack and sort by Status Code. Look for a 302 Found redirect, which indicates a successful login.

8. Login: Right-click the successful request, select "Show response in browser," and copy the URL to your browser to access the account.

Technical Explanation

The vulnerability exists because the backend uses a "short-circuit" logic. If the username doesn't exist, it returns an error immediately. If the username exists, it proceeds to the password hashing function (like Bcrypt), which introduces a measurable time delay. This creates a Side-Channel Attack opportunity where time itself acts as the data leak.

Conclusion

This lab demonstrates that security isn't just about what is in the response, but also how the response is delivered. Automated tools like Burp Intruder are essential for detecting these micro-timing differences that are impossible to spot manually.

How To Fix

°      Constant Time Processing: The application should ensure that the authentication process takes the same amount of time regardless of whether the username is valid or invalid.

°      Robust Rate Limiting: Do not rely solely on the X-Forwarded-For header for IP tracking, as it is easily spoofed. Use more reliable session identifiers or CAPTCHAs.

°      Hashing Consistency: Always perform a "dummy" hash comparison even if the username is not found to maintain timing consistency.
