Authentication | Practitioner

Lab Description

This lab uses a "stay-logged-in" cookie that is insecurely generated. The cookie contains the user's username and a hashed version of their password, all encoded in Base64.

To solve the lab, we need to brute-force the stay-logged-in cookie for the user carlos using the provided password list and access his account page.

Mindset & Strategy

When you see a cookie like d2llbmVyOjUxZGMzMGRkYzQ3M2Q0M2E2MDExZTllYmJhNmNhNzcw, the first step is to decode it. In this lab, the format is username:md5(password), then Base64 encoded.

Instead of manually hashing every password from the list, we will use Burp Intruder's Payload Processing Rules. This allows us to take a plain text password, hash it, add the prefix, and encode it on the fly for each request.

Step-By-Step-Execution

1. Capture the Request: Login with wiener:peter and check the "Stay logged in" box. In Burp Suite, capture the GET /my-account?id=wiener request. Observe the stay-logged-in cookie.

2. Analyze the Cookie: Send the cookie value to the Inspector or Decoder. You'll find it decodes from Base64 to: wiener:51dc30ddc473d43a6011e9ebba6ca770. The string after the colon is an MD5 hash of "peter".

3. Intruder Setup: Send the GET /my-account request to Intruder. Change the id parameter to carlos and set the payload position on the stay-logged-in cookie value.

4. Payload Configuration: Paste the provided password list into the Payloads tab. Now, we must transform these plain passwords into the correct cookie format using rules.

5. Payload Processing Rules: Add the following rules in order:
• Hash (MD5): Converts password to hash.
• Add Prefix (carlos:): Important to include the colon.
• Encode (Base64): Encodes the entire string.

6. Start Attack: Run the attack and sort by Length or Status Code. Look for a 200 OK response with a significantly different length.

7. Verification: Once you find the correct request, right-click and select "Show response in browser." Copy the URL into your browser.

8. Success: You are now logged in as Carlos. The lab is successfully solved through cookie manipulation.

Technical Explanation

The vulnerability here is Insecure Persistent Storage of credentials. By using a predictable hashing algorithm (MD5) and a static format (username:hash) in a client-side cookie, the application allows an attacker to forge valid session tokens. Even though the password is hashed, it is not "salted," making it vulnerable to simple brute-force or rainbow table attacks.

How To Fix

°      Use Opaque Tokens: Instead of storing credentials in the cookie, use a random, high-entropy session ID that maps to a server-side session.

°      Avoid Client-Side Hashing: Never store sensitive data like MD5 hashes in cookies, as they can be easily manipulated or cracked.

°      Digital Signatures: If data must be stored in a cookie, use a HMAC (Hash-based Message Authentication Code) with a secret server-side key to prevent tampering.
