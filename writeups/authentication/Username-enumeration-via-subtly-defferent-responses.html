Lab Description

This lab is subtly vulnerable to username enumeration and password brute-force attacks. It has an account with a predictable username and password, which can be found in the following wordlists:

°      Candidate usernames

°      Candidate passwords

To solve the lab, enumerate a valid username, brute-force this user's password, then access their account page.

Mindset & Strategy

When an application says "Invalid username or password," it’s doing its job well by not giving away which part of the credentials was wrong. However, sometimes developers make a mistake in the backend logic where a valid username triggers a slightly different response than an invalid one—even if the message on the screen looks identical.

Our strategy is to act like a detective looking for "micro-differences." We aren't just looking for a different message; we are looking for different response headers, hidden tags, or even a single missing period in a sentence. By using Burp Intruder with Grep-Match, we can automate the process of checking hundreds of usernames to see which one fails to trigger the standard "Invalid" error message.

Step-By-Step-Execution

1. Intercept & Analyze: Open the login page and submit a random username and password. Capture this POST request in Burp Suite.

2. Initial Testing: Send the request to Repeater. Try submitting the same request several times to check for rate limiting. In this lab, we notice there are no attempt limits, which clears the way for a brute-force attack.

3. Identify the Target String: Look closely at the response. We see the error message: <p class=is-warning>Invalid username or password.</p>. We will use this exact string to find our valid user.

4.Username Enumeration (Intruder):

°      Send the request to Intruder and set the payload position on the username parameter.

°      Paste the provided username list into the Payloads tab.

°      Go to Grep - Match under the Options (or Settings) tab. Clear the defaults and add the string: Invalid username or

       password.

°      Start the Attack.

5. Analyze the Results: Sort the results by the Grep-Match column. Most results will show a "1" (meaning the error message was found). Look for the one result that shows "0". This is our valid username because the server responded differently!

6. Password Brute-Force:

°      Take that valid username back to the Positions tab and fix it.

°      Set the payload position on the password parameter.

°      Paste the password list and start the attack.

7. Final Login: Look for a result with a 302 Redirect status code or a significantly different response length. Right-click that request, select "Show response in browser," and log in.

8. Result: Lab successfully solved.

Technical Explanation

The vulnerability here is Username Enumeration. Although the front-end message looks the same to a human, the backend is likely handling the logic in two different blocks: one for "user not found" and one for "user found, but password wrong." If there is even a tiny difference in how these two blocks render the HTML (like a hidden space or a different CSS class), an attacker can automate a script to distinguish between them.

Conclusion

This lab shows that "security through obscurity" (trying to hide the error) isn't enough if the underlying code isn't consistent. Subtle differences in responses are just as dangerous as clear error messages. As a pentester, always look at the raw response data in Burp, not just what appears in the browser window.

How To Fix

°      Uniform Error Messages: Ensure that the application returns the exact same HTTP response (including status codes, headers, and body length) regardless of whether the username or password was incorrect.

°      Account Lockout & Rate Limiting: Implement a mechanism to temporarily lock an account or block an IP address after a certain number of failed attempts. This makes automated enumeration much harder.

°      Bcrypt/Argon2 Consistency: Ensure that the time taken to process a "user not found" request is the same as a "wrong password" request to prevent Timing Attacks.
