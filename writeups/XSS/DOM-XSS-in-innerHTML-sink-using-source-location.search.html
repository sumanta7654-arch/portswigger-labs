Cross-Site Scripting | Practitioner

Lab Description

This lab contains a DOM-based cross-site scripting vulnerability in the search function. It uses an innerHTML assignment to display the search string. The innerHTML sink accepts HTML but does not execute <script> elements.

Goal: Perform a DOM XSS attack that bypasses the innerHTML restriction and calls the alert() function.

Mindset & Strategy

When dealing with innerHTML, a standard <script>alert(1)</script> payload will fail because modern browsers follow the HTML5 specification which prevents scripts inserted via innerHTML from executing.

Our strategy is to use an element that executes JavaScript as a side effect of rendering. The <img> tag with an onerror attribute is perfect for this. By providing an invalid src, we force the browser to trigger the onerror event handler immediately.

Step-By-Step-Execution

1. Initial Reconnaissance: Access the lab and perform a search for a simple string. Use your browser's developer tools (F12) to inspect how the search term is rendered in the DOM.

2. Testing Standard Payloads: Try entering <script>alert(1)</script>. You will notice the tag appears in the DOM, but no alert is triggered because of the innerHTML security behavior.

3. Crafting the Bypass: Since we cannot use script tags, we use an image tag. Enter the following payload into the search bar:
<img src=x onerror=alert("bypassed")>

4. Execution: Click the search button. The JavaScript on the page takes this string and assigns it to an element's innerHTML

5. Triggering the Sink: The browser attempts to render the <img> tag. Because src=x is not a valid image path, the onerror event is fired instantly.

6. Verification: An alert box with "bypassed" appears on the screen. The lab is successfully solved.

Technical Explanation

This is a DOM-based XSS vulnerability where location.search is the source and innerHTML is the sink. While innerHTML provides protection against <script> tags, it remains a dangerous sink because it still parses and renders other HTML elements. Event handlers like onerror, onload, and onmouseover allow an attacker to execute arbitrary JavaScript code by taking advantage of the browser's rendering engine.

How To Fix

°      Use Safer Sinks: Replace innerHTML with textContent. This ensures the browser treats the input strictly as text and never as HTML or executable code.

°      Sanitize Input: Use a library like DOMPurify to strip dangerous attributes (like onerror) before assigning the string to innerHTML.

°      Content Security Policy (CSP): Implement a CSP that disallows inline event handlers, which would prevent onerror from executing even if it is injected into the DOM.
