Cross-Site Scripting | Apprentice

Lab Description

This lab contains a stored cross-site scripting vulnerability in the comment functionality. Unlike reflected XSS, the malicious script is stored on the server's database and executed in the browser of anyone who views the blog post.

Goal: Inject a stored XSS payload that triggers an alert() box for anyone viewing the comments.

Mindset & Strategy

Stored XSS occurs when an application receives data from a user and includes that data within its later HTTP responses in an unsafe way. In a comment section, if the server doesn't sanitize the input, our <script> tag becomes a permanent part of the page's HTML.

The strategy is to find a form that saves data to the server—like a comment box—and see if the "nothing encoded" rule applies here as well. If it does, every visitor to this post becomes a victim of our script.

Step-By-Step-Execution

1. Select a Target Post: On the home page, click on any blog post to view its full content and the comment section below it.

2. Prepare the Payload: Scroll down to the "Leave a comment" section. In the comment text area, enter:
<script>alert("hacked")</script>

3. Submit the Data: Fill in the Name, Email, and Website fields with random information and click "Post Comment".

4. Trigger and Verify: Once the comment is posted, navigate back to the blog post or refresh the page. An alert box with "hacked" will appear immediately. The lab is now solved.

Technical Explanation

The vulnerability exists because the application performs No Input Validation and No Output Encoding. When the server retrieves the comment from the database to display it to users, it inserts the raw string directly into the HTML template.

Because the characters < and > are not converted to HTML entities, the browser's HTML parser treats the injected string as a legitimate script block. This is "Stored" because the attack persists even after the initial request is finished.

How To Fix

°      Context-Aware Output Encoding: Encode all user-supplied data before rendering it in HTML. < should become &lt;.

°      Sanitize HTML: If users are allowed to use some HTML (like bold or italics), use a library to "clean" the input and strip out dangerous tags like <script> or <iframe>.

°      Use Content Security Policy (CSP): Implement a strong CSP to prevent the execution of inline scripts and unauthorized external scripts.
