Cross Site Scripting | Medium

Lab Description

This lab contains a reflected cross-site scripting vulnerability in the search query tracking functionality.

The reflection occurs inside a JavaScript string with single quotes and backslashes escaped.

To solve this lab, perform a cross-site scripting attack that breaks out of the JavaScript string and calls the alert function.

Mindset & Strategy

When testing for XSS, our primary goal is to break out of the intended data context and execute arbitrary JavaScript. In this scenario, the application is performing some sanitization. If we try to use a single quote (') to break the string, the backend escapes it into \'. Similarly, a backslash (\) becomes \\.

Since the application effectively prevents us from escaping the JavaScript string itself, we need to shift our perspective. Instead of trying to manipulate the logic within the <script> tag, why not try to end the script block entirely? If we can force the browser to close the current script tag, we can then start a fresh one where no escaping rules apply to our payload.

Step-By-Step-Execution

1. Analyze the Reflection: Open the lab and perform a test search (e.g., test123). Right-click on the page, select "View Page Source", and find where your input is reflected. You will notice it sits inside a JavaScript variable within <script> tags.

2. Test the Filters: Try searching for ' or \. You will see that the application automatically adds a backslash to escape them, making standard payloads like '';alert(1)// useless.

3. The Breaking Point: Since we cannot break the string, we will target the HTML tag. In the search bar, enter the following payload: </script> <script>alert("hacked")</script>

4. Execute: Click search. The browser will interpret </script> as the end of the original script, and immediately execute our new <script> block.

5. Result: An alert box pops up with "hacked," and the lab is successfully solved.

Technical Explanation

The vulnerability exists because the application only filters characters that could break a JavaScript string (like quotes and backslashes) but fails to filter HTML tags.

°      </script>: This tells the browser's HTML parser that the current script section is finished.

°       <script>alert("hacked")</script>: This is a completely new, independent script. Because the browser has already "exited" the previous filtered context, it executes this new block without any escaping issues.

Conclusion

This lab teaches us an important lesson in web security: context matters. A developer might spend a lot of time securing a JavaScript string, but if they forget to validate the input against the broader HTML context, the application remains vulnerable. Effective defense requires looking at the entire environment where user input is rendered, not just the immediate variable it is assigned to.

How To Fix

To prevent this type of Reflected XSS, developers should implement the following:

°      HTML Encoding: All user-supplied data should be HTML-encoded before being rendered on the page. This would convert < into &lt; and> into &gt;, making the browser treat them as harmless text instead of executable tags.

°      Input Validation: Use a "whitelist" approach to only allow expected characters in the search bar.

°      Content Security Policy (CSP): Implement a strong CSP to restrict the execution of inline scripts, which would block this type of injection even if a tag is successfully closed.
