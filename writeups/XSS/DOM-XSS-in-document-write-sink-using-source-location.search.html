Cross-Site Scripting | Practitioner

Lab Description

This lab contains a DOM-based cross-site scripting vulnerability in the search query tracking functionality. It uses the JavaScript document.write function, which writes data to the page. The data is read from location.search, which you can control via the URL.

Goal: Perform a DOM XSS attack that breaks out of the existing img tag and calls the alert() function.

Mindset & Strategy

In DOM-based XSS, the payload is never sent to the server. Instead, it is processed entirely by the browser's JavaScript. The "Source" is location.search (the query string), and the "Sink" is document.write.

Our strategy is to inspect the JavaScript handling the search term. If the code places our input inside an attribute (like src), we need to "break out" of that attribute and the tag itself using characters like " and > before we can execute our own script.

Step-By-Step-Execution

1. Analyze the Page Source: Go to the lab and perform a search for a random string. Right-click and select "View Page Source". Look for the JavaScript responsible for the search tracking.

2. Identify the Vulnerable Code: You will find a script similar to:
document.write('<img src="/resources/images/tracker.gif?searchTerms='+query+'">');
This shows that our query is being placed directly inside the src attribute of an img tag.

3. Craft the Breakout: If we just type a script, it remains trapped inside the src quotes. To fix this, we start our payload with ">.
The first " closes the src attribute.
The > closes the img tag entirely.

4. Inject the Payload: Enter the following in the search bar:
"><script>alert("hacked")</script>

5. Execution: Press Enter. The browser's JavaScript will now write the following to the DOM:
<img src="/resources/images/tracker.gif?searchTerms=""><script>alert("hacked")</script>">

6. Verify Success: The original img tag is closed early, and your injected <script> executes immediately, triggering the "hacked" alert. The lab is solved.

Technical Explanation

This is a DOM-based XSS vulnerability because the flaw exists in the client-side script's handling of data. The document.write sink is dangerous because it treats the input as HTML code rather than plain text. Because the code does not sanitize or escape the " and > characters from the location.search source, an attacker can manipulate the DOM structure to execute arbitrary JavaScript[cite: 1].

How To Fix

°      Avoid Dangerous Sinks: Instead of document.write, use safer alternatives like textContent or innerText, which do not parse input as HTML.

°      Use Modern APIs: Use element.setAttribute() for modifying attributes, which automatically handles encoding of special characters.

°      Sanitization Libraries: If you must render HTML from a search query, use a library like DOMPurify to strip out dangerous tags and attributes before they reach the sink.
