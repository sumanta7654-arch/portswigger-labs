Cross-Site Scripting | Apprentice

Lab Description

This lab contains a simple reflected cross-site scripting (XSS) vulnerability in the search functionality. The application takes a search term and reflects it directly into the HTML response without any sanitization or encoding.

Goal: Perform a reflected XSS attack that calls the alert() function to prove the vulnerability exists.

Mindset & Strategy

When an application reflects your input back to the screen, the first thing to test is if the browser interprets that input as data or as code. By inputting HTML tags like <script>, we are testing if the server properly "escapes" special characters.

Since this lab has "nothing encoded," the browser will see our script tag and execute the JavaScript within it immediately upon loading the search results page.

Step-By-Step-Execution

1. Locate Input Field: Go to the lab's home page and find the search box. This is our primary entry point for the payload.

2. Inject Payload: In the search bar, type the following payload:
<script>alert("hacked")</script>

3. Trigger the Exploit: Press Enter or click the "Search" button. The page will reload, and you will see a pop-up alert box with the message "hacked". The lab is now solved.

Technical Explanation

The core issue here is the lack of Output Encoding. When you search for a term, the server sends back a response like: <h1>0 results for [YOUR_SEARCH_TERM]</h1>.

Because the server does not encode characters like < and > into &lt; and &gt;, the browser interprets our input as a valid script block rather than plain text. This is a classic Reflected XSS because the payload is "reflected" off the web server to the victim's browser.

How To Fix

°      Filter Input on Arrival: Implement strict allow-lists for user input to reject any unexpected characters or tags.

°      Encode Output on Display: Use context-aware encoding. Convert special HTML characters into their entity equivalents before rendering them in the browser.

°      Content Security Policy (CSP): Use CSP headers to restrict which scripts can execute on your page, effectively disabling inline scripts like the one used in this exploit.
