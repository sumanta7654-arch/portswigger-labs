Access Control | Practitioner

Lab Description

This lab contains a multi-step process for changing user roles. While the initial step is protected, the final confirmation step lacks proper access control. This allows a low-privileged user to skip the checks and perform administrative actions.

Goal: Exploit the unprotected confirmation step to promote your own user (wiener) to an administrator.

Mindset & Strategy

Developers often assume that users will only reach the final step of a process if they have successfully completed the previous steps. This is a Security through Workflow fallacy.

Our strategy is to capture the "Confirmation" request as an admin, which actually executes the role change, and then replay that specific request using a non-admin session cookie. If the backend doesn't re-verify permissions at the confirmation stage, we bypass the entire security check.

Step-By-Step-Execution

1. Capture the Final Action: Log in as administrator:admin. Go to the Admin Panel and promote carlos. When the confirmation prompt appears, click **Yes**. Capture this POST /admin-roles request in Burp Suite (look for confirmed=true).

2. Analyze in Repeater: Send this confirmation request to Repeater. Notice it contains the administrative action logic but is a separate request from the initial promotion selection.

3. Acquire Low-Privilege Session: Log out and sign in with your own account: wiener:peter. Use the browser inspector (F12 > Storage > Cookies) or Burp to copy your Session ID.

4. Session Hijacking in Request: Go back to Burp Repeater. Replace the Administrator's session cookie with your wiener session ID. Then, change the username parameter from carlos to wiener.

5. Execute Bypass: Send the modified request. The server should process this without verifying if wiener has permission, as it assumes the "Check" was already done in the previous step. You will see a 302 Found response.

6. Verify Status: Open the response in your browser. You will notice that wiener is now an administrator. The lab is successfully solved.

7. Technical Summary: This vulnerability is a classic Broken Access Control. Every request in a stateful transaction must be independently authorized. Relying on the user's progress through a UI workflow is never a substitute for server-side permission checks.

How To Fix

°      Continuous Authorization: Implement server-side access control checks at every single step of a multi-step process, especially on the final execution endpoint.

°      State Verification: Use cryptographically secure tokens or server-side session variables to track the state of a process, ensuring it cannot be bypassed or tampered with.

°      Role-Based Access Control (RBAC): Ensure that the promotion logic is strictly reserved for users with the 'Administrator' role at the API/endpoint level.
