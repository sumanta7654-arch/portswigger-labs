Access Control | Practitioner

Lab Description

This lab uses a front-end server to restrict access to the /admin path. However, the back-end server supports custom HTTP headers that can override the requested URL. We will exploit this discrepancy to bypass the access control list (ACL).

Goal: Use the X-Original-URL header to access the admin panel and delete carlos.

Mindset & Strategy

When you request /admin, the front-end (WAF/Proxy) sees the URL and blocks it immediately. But if we request the home page / and tell the server, "Hey, my original intent was to go to /admin" via a header, the front-end might let the request through because it only inspects the main URL path.

This is a Security Misconfiguration where the front-end and back-end don't agree on which part of the request defines the actual target resource.

Step-By-Step-Execution

1. Direct Access Failure: Try to visit /admin directly in your browser. You will receive an "Access denied" message. This confirms the front-end filter is working.

2. Test for Header Support: Open Burp Suite. Change the request to GET / and add the header X-Original-URL: /invalid. If you get a "404 Not Found", it proves the backend is trying to process the path provided in that header.

3. Access Admin Panel: Now, modify the header to X-Original-URL: /admin while keeping the main request path as GET /. Send the request and view the response in the browser. You should now see the admin dashboard.

4. Identify Delete Endpoint: Observe the "Delete" button for Carlos. Usually, it points to /admin/delete?username=carlos. Trying this normally will result in another "Access denied" from the front-end.

5. Craft Delete Exploit: In Burp, set the main request to GET /?username=carlos (sending the query parameter to the root). Then, add the header X-Original-URL: /admin/delete.

6. Execute Attack: Send this request. The front-end sees a request to / (allowed), but the back-end processes it as /admin/delete with the username parameter.

7. Verification: Check the response. A 200 OK or a success message indicates the user has been deleted. Refresh the lab page to confirm it is solved.

Technical Explanation

This vulnerability is caused by an Inconsistent Interpretation of HTTP Requests. The front-end server only checks the URL line of the HTTP request, while the back-end application framework (like certain versions of Symfony or Zend) allows the X-Original-URL or X-Rewrite-URL headers to override the routing logic. This discrepancy allows an attacker to "hide" the forbidden path from the security filter.

How To Fix

°      Disable Override Headers: Configure the backend server or application framework to ignore non-standard headers like X-Original-URL for routing purposes.

°      Normalize Requests: Ensure that both front-end and back-end servers use the same rules for parsing and authorizing URLs.

°      Defense in Depth: Apply access control logic at the application level (code) rather than relying solely on network-level filters (WAF/Proxy).
