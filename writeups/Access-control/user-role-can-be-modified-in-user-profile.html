Access Control | Practitioner

Lab Description

This lab allows users to update their own profile information. However, the profile update function is vulnerable to mass assignment, allowing a user to modify sensitive fields like their roleid.

Goal: Escalate your privileges by modifying your user profile JSON, access the admin panel, and delete carlos.

Mindset & Strategy

Many modern applications use JSON to send data to the backend. A common mistake developers make is using a "blind update" method—taking the entire JSON object from the request and saving it directly into the database.

If we can guess or discover hidden field names (like roleid, isAdmin, or permissions), we can include them in our update request. Even if those fields aren't visible on the UI, the backend might still process them and grant us higher privileges.

Step-By-Step-Execution

1. Baseline Analysis: Log in with wiener:peter. Notice that the initial login doesn't show any admin features or role-related parameters in the main navigation.

2. Identify Update Function: Go to the user account page and find the "Update Email" feature. Enter a new email and click update.

3. Capture Update Request: In Burp Suite, capture the POST /my-account/change-email request. You will see the data is sent in a JSON format: {"email":"test@example.com"}.

4. Exploit Mass Assignment: Send this request to Repeater. We will now attempt to inject a hidden parameter. Modify the JSON body to include the roleid field:
{"email":"wiener@normal-user.net", "roleid": 2}

5. Verify Elevation: Send the modified request. If the server responds with 200 OK, refresh your page in the browser. You should now see an Admin Panel link available to you.

6. Access Admin Panel: Navigate to the admin panel. Since your roleid is now 2 (which typically represents 'Administrator' in many systems), the server grants you full access.

7. Delete Target: Locate the user carlos and click Delete. The request will now be authorized by your escalated session, and the lab will be solved.

Technical Explanation

The vulnerability is Mass Assignment. This happens when an application's backend framework automatically maps request parameters to internal database models without a "whitelist" of allowed fields. By adding "roleid": 2 to the JSON, the attacker successfully overwrites their own permission level because the backend code trusts the input content implicitly.

How To Fix

°      Use Whitelisting: Explicitly define which fields the user is allowed to update (e.g., only email and display_name). Reject or ignore any other keys in the JSON.

°      DTOs (Data Transfer Objects): Implement DTOs to separate the data coming from the user from the actual database model. Never pass raw user input directly into a database update function.

°      Read-Only Fields: Ensure that sensitive fields like roleid can only be modified by specific, highly-privileged administrative endpoints.
