Access Control | Apprentice

Lab Description

This lab is vulnerable to horizontal privilege escalation. The user profile page is retrieved based on a id parameter in the URL, but the server fails to check if the requesting user has permission to view that profile.

Goal: Access carlos's account page, retrieve his API key, and submit it to solve the lab.

Mindset & Strategy

Horizontal Privilege Escalation happens when a user can access data belonging to another user with the same level of permissions. This is often due to an Insecure Direct Object Reference (IDOR).

The strategy is to find a unique identifier in the URL (like a username or UUID) and simply swap it with another user's identifier. If the backend doesn't validate the session against the requested ID, we can "impersonate" any user just by changing the URL.

Step-By-Step-Execution

1. Analyze Your Account: Log in with wiener:peter. Navigate to your account page. Notice the URL structure: /my-account?id=wiener. You can see your own API key on this page.

2. Manipulate the Parameter: Click on the browser's address bar. Manually change the value of the id parameter from wiener to carlos (i.e., /my-account?id=carlos) and press Enter.

3. Extract Data: Because the application lacks proper authorization checks, you are now viewing Carlos's account page. Locate his API Key displayed on the screen and copy it.

4. Submit Solution: Click the "Submit solution" button at the top of the lab banner. Paste the copied API key into the input field and click OK.

5. Lab Solved: Once the correct API key is submitted, the lab will be marked as completed. You have successfully performed an IDOR attack to steal sensitive information from another user.

Technical Explanation

The vulnerability exists because the application performs Authentication (checking who you are) but skips Authorization (checking what you are allowed to do). The server-side code likely fetches the user data based on the id parameter directly from the database without verifying if that id matches the user's current session token.

How To Fix

°      Session Validation: Instead of relying on a URL parameter, the application should retrieve the user's ID directly from their secure session object on the server.

°      Access Control Checks: If a parameter must be used, the backend must verify that the requested id belongs to the currently logged-in user before returning any sensitive data.

°      Use Non-Predictable IDs: Using random UUIDs instead of simple usernames makes it harder for an attacker to guess other valid identifiers, though this is "defense in depth" and not a primary fix.
