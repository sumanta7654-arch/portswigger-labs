Access Control | Practitioner

Lab Description

This lab is vulnerable to horizontal privilege escalation. When a user attempts to access another user's profile, the application initiates a redirect to the login page. However, it still includes the target user's data in the response body before the redirect happens.

Goal: Access carlos's account page, find the leaked API key in the HTTP response, and submit it.

Mindset & Strategy

Sometimes, developers implement security checks that only "look" secure. In this case, the server sees an unauthorized request and says, "Go to the login page" (a 302 Redirect). But the server-side code is poorly written—it fetches the user's data first and attaches it to the response body before sending the redirect command.

Our strategy is to use Burp Suite to catch that "in-between" response. While your browser automatically follows the redirect and shows the login page, Burp Suite allows us to inspect the original response containing the leaked data.

Step-By-Step-Execution

1. Initial Login: Log in with wiener:peter. Navigate to your account page. Observe the URL: /my-account?id=wiener.

2. Attempt Access: Manually change the URL parameter to id=carlos. Notice that the browser immediately kicks you back to the /login page. This looks like a successful security measure at first glance.

3. Analyze with Burp: Go to Burp Suite's HTTP History. Look for the GET /my-account?id=carlos request. You will see a 302 Found status code.

4. Inspect Response Body: Click on that 302 request and look at the Response tab. Scroll down past the headers. You will see the full HTML of Carlos's account page, including his **API Key**, hidden right there in the body!

5. Solve the Lab: Copy the leaked API key from the Burp response. Go back to the lab, click "Submit solution", and paste the key. The lab is now solved.

Technical Explanation

This is a classic Information Disclosure vulnerability. The server's logic is executing in the wrong order: it generates the sensitive content (render_profile_page) and then realizes the user isn't authorized (redirect_to_login). Because the content is already in the buffer, it gets sent along with the redirect header. A secure application should terminate the request immediately without sending any profile data if authorization fails.

How To Fix

°      Early Authorization: Perform authorization checks at the very beginning of the request handling process, before any sensitive data is retrieved or rendered.

°      Atomic Redirects: When sending a redirect header (like Location: /login), ensure the response body is empty or contains only a generic message. Use exit() or return immediately after the redirect command.

°      Secure Coding Practices: Avoid "leaky" logic where error messages or redirect responses provide more information than necessary to the client.
