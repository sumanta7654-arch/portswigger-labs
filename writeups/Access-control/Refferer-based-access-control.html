Access Control | Practitioner

Lab Description

This lab controls access to administrative functions based on the Referer header. By spoofing this header, a regular user can bypass access controls and perform sensitive actions.

Goal: Use the wiener user session to promote yourself to an administrator by manipulating the Referer header in the request.

Mindset & Strategy

The Referer header is an HTTP request header that identifies the address of the web page from which the link to the currently requested page was followed. Some applications incorrectly trust this header to verify that a request originated from a trusted page (like the admin panel).

Since the Referer header is sent by the browser, it is entirely under the user's control. We can use Burp Suite to change its value and trick the server into thinking we are performing the action from the authorized administrator dashboard.

Step-By-Step-Execution

1. Analyze Admin Flow: Log in as administrator:admin. Go to the Admin Panel and promote carlos. In Burp Suite, capture the GET /admin-roles?username=carlos&action=upgrade request and send it to Repeater.

2. Identify the Security Check: Notice the Referer: [URL]/admin header in the request. The server likely checks this header to decide whether to allow the role change.

3. Acquire Victim Session: Log out and sign in as wiener:peter. Use the browser inspector or Burp Suite to get your wiener session cookie.

4. Initial Bypass Attempt: In Burp Repeater, replace the Administrator's session cookie with your wiener session cookie. If you try to access /admin directly, the server will tell you that the interface is only for administrators.

5. Spoofing the Referer: Go back to the captured /admin-roles request in Repeater. Change the username parameter to wiener and ensure your wiener session cookie is in place.

6. Execute Attack: Ensure the Referer header is set to the admin panel URL (e.g., https://[lab-id].web-security-academy.net/admin). Send the request. The server sees the trusted Referer and allows the action.

7. Verification: You should see a 302 Found response. Refresh the page in Wiener's browser. You are now an administrator, and the lab is solved. This proves that Referer headers should never be used for security-critical authorization.

Technical Explanation

This vulnerability is a classic case of Insecure Authorization based on untrusted metadata. The application's security logic is flawed because it assumes that if a request originates from /admin (as indicated by the Referer header), the user must already be an administrator.

Since the Referer header is a client-side component, it can be easily manipulated using tools like Burp Suite. When we manually set the Referer to the admin path, we effectively bypass the "Where did this request come from?" check without ever having to prove "Who is making this request?" via a secure session role check.

How To Fix

°      Never Trust Headers: Never use client-side headers like Referer or User-Agent for access control decisions.

°      Session-Based Authorization: Check the user's role and permissions directly from the secure, server-side session data before executing any privileged function.

°      CSRF Protection: Use anti-CSRF tokens to ensure requests are intentional and come from authorized sessions, rather than relying on weak header checks.
