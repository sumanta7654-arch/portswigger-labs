Access Control | Practitioner

Lab Description

This lab implements access controls based partly on the HTTP method of the request. You can bypass these controls by using a different HTTP method to perform privileged actions.

Goal: Use a regular user session (wiener) to promote carlos to an administrator by bypassing the method-based restriction.

Mindset & Strategy

Developers often assume that sensitive actions (like promoting a user) will only ever be sent via a POST request. They might configure their security rules to block POST /admin-roles for non-admin users, but forget to block GET /admin-roles.

By switching the request method and moving the parameters from the body to the URL (Query String), we can often slip past these narrow security filters.

Step-By-Step-Execution

1. Capture Admin Request: Log in as administrator:admin. Go to the Admin Panel and promote carlos. Capture this POST /admin-roles request in Burp Suite and send it to Repeater.

2. Get Non-Admin Session: Open a private/incognito window and log in as wiener:peter. Open Developer Tools (F12) > Storage > Cookies and copy the session ID for Wiener.

3. Test for Bypass: In Burp Repeater, replace the Administrator's session cookie with Wiener's session cookie. Send the request. You will receive an "Unauthorized" or "Access Denied" error because POST is restricted.

4. Switch Method: Right-click the request in Burp and select "Change request method". This will convert the POST to a GET request.

5. Adjust Parameters: In a GET request, the body is ignored. You must move the parameters to the URL.
Original: username=carlos&action=upgrade in body.
Modified: /admin-roles?username=wiener&action=upgrade in the GET path.

6. Execute Exploit: Ensure Wiener's session cookie is still there and send the request. You should see a 302 Found redirect, indicating success.

7. Verification: Refresh the page in Wiener's browser. If the exploit worked, Carlos will now have administrative privileges and the lab will be solved.

Technical Explanation

This vulnerability occurs because the application's security framework is configured to perform Method-Specific Authorization. Instead of checking if "User X can access Function Y," it checks if "User X can use Method Z on Function Y." This is a flawed approach. Attackers can often use alternative methods like GET, HEAD, or even POST (if it was a GET-only check) to trigger the same backend logic while bypassing the firewall.

How To Fix

°      Method-Agnostic Access Control: Ensure that authorization checks are applied to the resource or function itself, regardless of which HTTP method is used to access it.

°      Strict Method Enforcement: If an endpoint is only supposed to accept POST, explicitly reject all other methods at the routing level with a 405 Method Not Allowed response.

°      Standardized Routing: Use modern frameworks that automatically handle session validation before any business logic is executed, ensuring uniform security across all request types.
