Access Control | Apprentice

Lab Description

This lab hides its administrative console at an unpredictable URL. However, the application leaks this URL to all users through client-side scripts.

The goal is to find the hidden URL by inspecting the source code and use it to delete the user carlos.

Mindset & Strategy

Developers often write JavaScript to dynamically change the UI based on user roles. If this logic is handled on the front-end (browser) rather than the back-end (server), any user can see the "hidden" logic by simply reading the source code.

Even if the URL is a random string like /admin-tfisdl, it is still "public" if it resides in a script tag that the browser must download to render the page.

Step-By-Step-Execution

1. Inspect Source: Access the lab's home page. Right-click anywhere on the page and select "View Page Source" (or press Ctrl+U).

2. Analyze JavaScript: Scroll down to the bottom of the HTML. You will find a <script> block containing logic for the admin panel. Look for a line like: adminPanelTag.setAttribute('href', '/admin-tfisdl');.

3. Access Hidden Panel: Copy the revealed path (e.g., /admin-tfisdl) and paste it into the URL bar after the main lab address. This will take you directly to the unprotected admin dashboard.

4. Perform Action: Locate the user carlos in the list and click Delete. The lab will be marked as solved once the user is removed from the database.

Technical Explanation

This is a classic Information Leakage vulnerability. The server-side code is trusting the client-side code to "hide" the admin link. Since attackers have full control over the client environment (the browser), they can inspect every line of HTML and JS. Randomizing a URL (Unpredictable URL) is not a substitute for proper Access Control.

How To Fix

°      Server-Side Authorization: Always check the user's session and permissions on the server before rendering any administrative links or processing admin requests.

°      Remove Client-Side Logic: Sensitive URLs and administrative logic should never be sent to a non-admin user's browser, even in hidden script tags.

°      Secure Design: Use standard paths (like /admin) but protect them with multi-factor authentication and robust role-based access controls (RBAC).
