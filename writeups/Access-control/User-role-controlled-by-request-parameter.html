Access Control | Practitioner

Lab Description

This lab uses a request parameter to determine the user's privilege level. By manipulating this parameter, we can escalate our privileges from a regular user to an administrator.

The goal is to access the admin panel and delete the user carlos by spoofing our administrative status.

Mindset & Strategy

Web applications should always manage user roles on the backend. If you see a parameter like admin=false or role=user in a URL or a Cookie, it’s a massive red flag. It suggests that the server is asking the browser, "Are you an admin?" instead of checking its own secure database.

Our strategy is simple: **Intercept and Lie**. We will tell the server that we are an administrator by changing the boolean value from false to true.

Step-By-Step-Execution

1. Capture Account Request: Log in with wiener:peter. Open Burp Suite and capture the GET /my-account?id=wiener request. Notice the interesting parameter in the cookie or URL: Admin=false.

2. First Manipulation: Change Admin=false to Admin=true. Send the request to the browser. You will notice that a new link for the "Admin Panel" appears in the navigation bar.

3. Access Admin Panel: Click on the Admin Panel link. Capture this GET /admin request in Burp Suite before it reaches the server.

4. Bypass Panel Restriction: Send the request to Repeater. If you try to access it normally, you might get a "401 Unauthorized" because the server checks the parameter again. Modify it to Admin=true once more.

5. Analyze Admin Dashboard: After sending the modified request, view the response in the browser. You should now see the list of users and the "Delete" buttons for each.

6. Target Carlos: Locate carlos and click the Delete button. Immediately capture the resulting request (e.g., GET /admin/delete?username=carlos).

7. Final Exploit: In Burp Suite (Repeater or Proxy), ensure that the Admin=true parameter is still present in this delete request. Send the request.

8. Verification: Check the response. If you see a 302 Found or a success message, the user has been deleted. Refresh the page to see the "Lab Solved" banner.

Technical Explanation

The core vulnerability is Improper Authorization. The application logic trusts user-supplied input to define security clearance. Since HTTP requests are entirely under the attacker's control, any "security check" that happens on the client-side or depends on client-side parameters is essentially non-existent. This allows for horizontal and vertical Privilege Escalation.

Conclusion: Never trust the client. Authorization checks must be performed on the server-side using secure session data that the user cannot modify directly.

How To Fix

°      Server-Side Session Management: Store the user's role in a secure, server-side session object (e.g., in Redis or a database) rather than in a request parameter or a modifiable cookie.

°      RBAC (Role-Based Access Control): Implement a central authorization module that checks the user's role against the required permission for every sensitive URL or action.

°      Principles of Least Privilege: Ensure that by default, users have no administrative access, and access is only granted based on internal server records.
