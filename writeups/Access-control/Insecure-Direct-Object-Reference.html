Access Control | Practitioner

Lab Description

This lab stores user chat transcripts directly on the server's file system and retrieves them using static filenames. It fails to implement proper access controls, allowing any user to view transcripts belonging to others.

Goal: Find carlos's password in a leaked chat transcript and use it to log in to his account.

Mindset & Strategy

IDOR occurs when an application provides direct access to objects based on user-supplied input. In this case, the object is a .txt file. If we see a pattern like 1.txt, 2.txt, it is highly likely that 1.txt belongs to another user or session.

By simply decrementing or guessing the numeric filename in the request, we can bypass the UI and talk directly to the server's storage to extract private conversations.

Step-By-Step-Execution

1. Generate Traffic: Access the lab and go to the Live Chat section. Send a few messages and then click on the "View transcript" button. A file (e.g., 2.txt) will be downloaded to your system.

2. Intercept Request: In Burp Suite, locate the GET /download-transcript/2.txt request in your HTTP history. Send this request to Repeater.

3. Manipulate ID: In Repeater, change the filename from 2.txt to 1.txt. Send the modified request. Look at the response body—you will see a chat transcript between Carlos and a support agent.

4. Extract Password: Read through the transcript carefully. Carlos likely shared his password with the agent. Copy the password (e.g., zjo406yppp9tox0q6a9n).

5. Account Takeover: Go to the Login page. Use the stolen credentials: carlos:zjo406yppp9tox0q6a9n. Once logged in, the lab will be marked as solved.

Technical Explanation

The vulnerability exists because the application uses a Direct Object Reference (the filename) to retrieve data without an Authorization Layer. The server assumes that if a user knows the filename, they must have the right to see it. A secure implementation should map these files to specific user sessions or use non-guessable, random identifiers (UUIDs) while still verifying ownership on the server side.

How To Fix

°      Indirect Object References: Map internal files to temporary, session-specific keys that only work for the authorized user.

°      Access Control: Before serving a file, the server must check if the authenticated user has the permissions required to access that specific resource.

°      Non-Predictable Names: Use long, random strings or GUIDs for filenames so that they cannot be guessed by incrementing numbers.
