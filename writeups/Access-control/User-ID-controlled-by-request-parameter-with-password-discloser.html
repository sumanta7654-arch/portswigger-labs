Access Control | Practitioner

Lab Description

This lab contains an IDOR vulnerability where the user's account page reveals their password. By manipulating the id parameter, we can access the administrator's account page and steal their credentials.

Goal: Find the administrator's password, log in as administrator, and delete carlos.

Mindset & Strategy

Developers sometimes include "hidden" input fields in HTML to store user data for processing. While these aren't visible on the rendered page, they are fully visible to anyone who looks at the Page Source.

Our strategy is to exploit the lack of access control to view the administrator's account page and then inspect the raw HTML to find their plaintext password.

Step-By-Step-Execution

1. Baseline Check: Log in with wiener:peter. On your account page, right-click and select "View Page Source". You will notice your password peter is stored in a hidden input field.

2. Access Administrator Profile: In the URL bar, change the id parameter from wiener to administrator (e.g., /my-account?id=administrator) and press Enter.

3. Extract Credentials: Once on the administrator's account page, view the page source again (Ctrl+U). Search for the password field. You will find the administrator's password (e.g., 89spkdo56oogqf5t1p2i).

4. Account Takeover: Log out of your current session. Go to the login page and sign in using the stolen credentials: administrator:[stolen_password].

5. Navigate to Admin Panel: Now that you are logged in as the real administrator, you will see the Admin Panel link in the header. Click on it to access the user management dashboard.

6. Delete Carlos: Find the user carlos in the list and click the Delete button next to their name.

7. Lab Solved: The application processes the deletion request because you are now an authenticated admin. The lab is successfully completed.

Technical Explanation

This vulnerability is a combination of Horizontal Privilege Escalation (viewing another user's profile) and Sensitive Data Exposure (storing passwords in HTML). Even if the password is "hidden" via CSS or input types, it is still sent to the client's browser, meaning it is compromised. A secure system should never send a password back to the user's browser after the initial login.

How To Fix

°      Server-Side Authorization: Verify that the id in the URL matches the logged-in user's session ID before rendering the page.

°      Data Protection: Never include sensitive information like passwords or PII (Personally Identifiable Information) in the HTML source code, even in hidden fields.

°      Secure Password Management: If a user needs to change their password, provide a dedicated form that doesn't reveal the current password.
