Access Control | Practitioner

Lab Description

This lab is vulnerable to horizontal privilege escalation via an IDOR vulnerability, but it uses unpredictable Globally Unique Identifiers (GUIDs) for user IDs. To exploit this, we must first discover another user's GUID.

Goal: Find carlos's GUID, access his account page, and retrieve his API key.

Mindset & Strategy

Using unpredictable IDs like GUIDs (e.g., 035aefad-8ab2...) adds a layer of security through obscurity. It prevents an attacker from simply guessing the next ID. However, if the application displays these IDs anywhere in the public interface, they lose their protective value.

Our strategy is Reconnaissance. We will look for any part of the website where Carlos's activity is shown (like a blog post) to see if the application reveals his unique ID in the background or the URL.

Step-By-Step-Execution

1. Baseline Check: Log in with wiener:peter. Notice your URL is /my-account?id=[YOUR-GUID]. If you try to change the ID to something random, the server redirects you back to the login page, proving you need a valid GUID.

2. Finding the Target GUID: Go back to the home page. Find a blog post written by carlos and click on his name to view his profile. Look at the URL; it will look like /blogs?userId=035aefad-8ab2-4c13-ac63-24825dc00a9e. Copy this GUID.

3. Execute IDOR: Go back to your "My Account" page. In the address bar, replace your own GUID with the one you just copied from Carlos's profile. Press Enter.

4. Data Extraction: You are now successfully viewing Carlos's account page. Copy the **API Key** displayed on the screen.

5. Submit & Solve: Click "Submit solution" in the lab banner, paste the API key, and click OK. The lab is now solved. You have demonstrated that unpredictable IDs are not a valid defense if they are leaked elsewhere.

Technical Explanation

The vulnerability here is still **Insecure Direct Object Reference**. Although the ID is hard to guess, the server still doesn't check if the GUID in the URL matches the one stored in the user's session. The application blindly trusts that anyone who knows the GUID must be authorized to see that account, which is a flawed assumption in a public-facing system.

How To Fix

°      Server-Side Authorization: Always verify that the requested resource ID matches the ID of the authenticated user in the current session.

°      Avoid ID Leakage: If sensitive actions are tied to a GUID, ensure that the GUID isn't exposed in public areas like blog posts or profile URLs unless absolutely necessary.

°      Use Indirect References: Map internal IDs to temporary, session-specific "aliases" so that the real database ID is never exposed to the client.
